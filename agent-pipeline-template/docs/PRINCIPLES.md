# Principles & non-goals

> **[TEMPLATE — fill this in]**
> This is the **operating contract**: the principles the project holds to, and the
> things it deliberately does *not* do. Where `PURPOSE.md` is the *why* and
> `STRATEGY.md` is *what we build right now*, this is the ***how we behave*** — the
> standing rules a change is judged against, whoever (or whatever) makes it.
>
> It is the single most valuable doc for an autonomous pipeline: Stage A checks a
> proposal against these principles and non-goals, Stage B keeps builds faithful to
> them, and an automated reviewer (if you add one) scores work against this exact list.
> Goals say what to aim for; principles say where the edges are. Two references worth
> stealing from: Marty Cagan's *product principles* (a named artifact, separate from
> strategy) and Ray Dalio's *Principles* (codified, actionable decision rules).
>
> Write:
> - **A litmus test.** The one question every change must answer (often lifted straight
>   from `PURPOSE.md`).
> - **Principles.** 5–10 durable, *actionable* rules, each stated so an agent can apply
>   it across many situations (e.g. "when in doubt, remove a choice rather than add
>   one"). Avoid platitudes — each one should be able to *reject* a real proposal.
> - **Non-goals.** The things you deliberately don't do right now — the hard "no"s. A
>   proposal that depends on one is out of scope regardless of quality. Say whether each
>   is permanent or just "not this phase."
> - **A tie-breaker.** When two principles conflict, which wins? An agent needs a
>   priority order, not just a list.
